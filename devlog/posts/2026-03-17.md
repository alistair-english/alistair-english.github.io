# Harry Potter and the chamber of accepting a non-linear git history

I ranted in much detail last week about linear git history and stuff. I thought it was interesting to share what I decided with the benefit of some distance over a long weekend.

I made one key claim on friday:

> I am happy to relinquish "merge" PRs, and thus full branch history, in return for automatic enforcement of "linear history" - i.e. not having people merge main back into their feature branch to get their feature branch up to date. The result is we end up with "merge commits" - where we squash the feature branch into one commit on main.

In short, I changed my mind. The main reason is that when testing stuff in the field, we tend to use commits as a bit of a testing log - e.g. for some parameter value. That history is valuable, but also having the commits around it showing what we changed is also valuable. I don't think that githubs inability to build this feature means we should get rid of that to just get an auto-enforcement feature. Plus, i haven't looked into it but if it becomes a problem i'm sure there are other ways around it, like git hooks or github actions to check these things.

That is all.

# memoirs of a vibecoder

Over the weekend I had fun playing with my new ralphlooping buddy, claudepad. I don't feel like I fully understand what the hell is going on yet to say anything very intelligent about it, but I'm starting to see where the "holy shit" people are coming from. The tldr of the stuff i've been playing with is a system set up as follows:
- a `specs/` folder, with new changes to be made - these really are "specs", i.e. specifiying outcomes, not how to do it
- a "planning" prompt, which is quite detailed - explaining: 1. get oriented with the codebase, 2. read through all the specs, 3. break the specs into self-contained implementation steps that can be built, tested and committed together _and crucially_ can be (as best you can estimate) fit into about 50% of the context window of a fresh instance of claude code
    - this planning prompt should create `implementation_plan.md` - a file with a list of steps, each with an unmarked checkbox
- a "build prompt" - the prompt is: 1. get oriented with the codebase, 2. take the top item off the implementation plan, 3. implement it, 4. _test it_ - this is the most important, and where you need to spend the most human time thinking (the "ralph looping" people call it "backpressure"), 4. commit it and mark it done on the list
- a bash script that runs in a loop, spinning up a new instance of claude code and feeding it the prompt. The whole point is that when a feature is done, CC exits, the context is nuked and a new instance comes in a starts fresh - amnesia by design (apparently to prevent "context rot").

Using this system I made a lil nextjs app that you track your whiskey with. You can input your collection of bottles in and then whenever you do a "tasting" of a bottle, you can log your tasting notes. Its deployed on vercel, uses supabase for auth and db, and shadcn for ui stuff.

I feel a little scared posting the url publicly but, I think its worth a click around to give some context to this post. So yolo, check it out here (at the time of writing its free to setup an account): https://peatfolio.vercel.app/

Some surprising things:
- aside from creating a vercel and supabase account, and then logging into the cli - i didnt do any deployment stuff, claude was able to handle project configuration via the cli
- this cli was also surprisingly useful for debugging production issues too - I was able to say "im getting a 500 error in prod, pull the logs and tell me whats wrong", and i got the right fix
- the app works (this is and isn't surprising, but there is more than one page, with linked user flows between concepts - you can create a tasting from the "tastings" page, but also via clicking on a bottle and then saying "add" tasting)
- The app was completely bootstrapped from zero using ralph loops
    - For the initial setup, i got claude to "interview me" to flesh out the idea, then write a PRD
    - Then i used another instance to convert the PRD into specs, and then did the plan and loop thing and it didn't completely go to shit
        - I did wake up to an error page initially (after ralph looping all night), but this was due to the fact that I didn't have any "runtime" testing set up - I only had code linting and unit tests
        - also to be fair, if i recall correctly the issues on deployment were more configuration things (missing the db connection environment var)
- skills are cool
- i didn't actually manually set up any of the ralph loop system, in a very meta move, I got claude to research ralph loops, summarise its findings into a skill and then used that skill to set up a ralph loop system in the repo
- the workflow is hella fun - most of my interactions go something like this:
    > I've found issue xyz _or_ I want abc new feature - how should we do this / research the best way to fix this etc. Then write a /ralph-loop spec to address it.
    
    after some back and forth (or maybe it just gives a good answer up front) and the specs are done, i just let it rip with a plan and then call `./loop.sh` and come back later

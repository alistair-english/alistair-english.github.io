# Dockerception

A problem I've been working on today:

Currently, we use docker devcontainers to manage our ros environment. We build off the osrf base images, install some of our more gnarly system dependencies and then do a decent amount of fucking about to get source mounted into the containers properly, builds cached, user uids correctly set so that source edits dont come through as root, external devices come through properly, gui apps work... you get the idea. It works though. And the user experience isn't terrible, particularly when running through vscode. There are some annoying things however - most of them stemming from being inside a container lol. 

This is all worth it because ROS environments tend to be slippery little fishes, and so ensuring we have consistent dev and "deploy" environments on our actual tractor means we are yet to have any weird build/runtime issues due to inconsistent environments, which is nice.

However, I've been starting to put some effort into getting good integration tests happening across our stack. We're on a big push at the moment to start getting the system more robust, as we move to more on-client testing. There are some good packages out there for integration testing multiple nodes together (`launch_testing`), but a major limitation is that if you run multiple tests at once, you risk cross-talk, because everone is on the same dds network. This is quite annoyoing, because then you can't _really_ parallelise your tests. There are some partial solutions, like shims that give each test a different ROS_DOMAIN_ID - but this feels fragile to me. If only there was a lightweight way to spin up multiple isolated environments...

A cool project i've been sitting on for a while without a real push to use is (Dagger)[https://dagger.io]. The tldr is it lets you script your build / test / ci systems, by giving you primitives to spin up and down containers. Sounds weird, im probably explaining it badly - go watch their video or something. Anyway, i _think_ this would be a good candidate for orchestrating something like a parallel integration test setup.

Only problem is, our dev environment is already inside a container - and unlike thanos, I can't use the containers to run the containers (I really should set up images on here so i can post memes). 

Anyway this leads me to a secondary cool thing ive been sitting on without a motivating usecase - using nix for packaging ROS. I haven't got a full concept scoped out yet, but I've been doing a lot of claude-assisted research today, to get my head around what exactly a nix-based ros dev environment would look like and how that might free us up to use containers as a tool, instead of our operating environment.

Nix (well, specificially NixOs)... ok actually we should explain the nixes real quick. _Nix_ is a package manager and build system that builds each and every package in isolation. What does this mean? I dunno man. The way i heard it explained was - start with some tool you want on your system, say GCC. Now, GCC has linked dependencies, and compiled source (i suppose, idk how gcc works). Now, imagine for a second that GCC _only_ had compiled source - no dependencies. Instead of versioning gcc via some manual version number that a human operates, what if when we built gcc, we gave it a "version" that was a hash of all its input source. Now, consider that we also do this with GCCs linked dependencies (brb, let me google a gcc dependency for this explanation) - ok zstd i guess (don't @ me, i just saw it on the page and it sounds about right). Now zstd, built from source, gets a version that is a hash of the input files and now GCC - built from source _plus_ a particular version of zstd gets a version that is some hashing function (don't worry about it because i dont know what it is) with inputs of the source files + the zstd version. The cool thing about this is that you can now have any set of packages with any set of dependencies you like. Its also quite space efficient (kinda like how linking is) because now if you have some other project that also wants that version of zstd that we got earlier, it can also reference it, knowing that it is the right thing. So yeah, _nix_ is the system that makes all of this happen. And its declarative, because you just say "i want xyz package" and using this whole hash shenanaigans thing go figure out what needs to be gotten to make that all work.

Now of course, this would be very terrible if you had to build every package from source, but luckily there are some nerds out there on the internet that do a good job of running _nixpkgs_ - which hosts prebuild versions of the packages with the versioning thing we talked about above. Basically instead of compiling things, you can download the "version" you need.

Finally, we have _NixOS_ which is a whole ass operating system built on top of _nix_. Theres the package bit, but then they've also managed to make the rest of the operating system declarative as well. In general you don't ever run a stateful configuration command - you update a config file and then "rebuild" your system with the new config.

## what the hell are you talking about

yeah ok fair - heres the tldr from the nixos [website](https://nixos.org/):

**Reproducible**: Nix builds packages in isolation from each other. This ensures that they are reproducible and don’t have undeclared dependencies, so if a package works on one machine, it will also work on another.

**Declarative**: Nix makes it trivial to share development and build environments for your projects, regardless of what programming languages and tools you’re using.

**Reliable**: Nix ensures that installing or upgrading one package cannot break other packages. It allows you to roll back to previous versions, and ensures that no package is in an inconsistent state during an upgrade.

Somehow, this set of attributes is the perfect cocktail for letting an LLM go absolutely ham on a system and getting outcomes that are good? Also, claude is somehow cracked at nix.

In effect, the main magic juice is the declarative bit. It has two advantages: 1. you can derive the state from the file (good for an aGeNtIc WoRkFlOw - just read the file and you know whats happening - don't have to query anything) and 2. its only "action vectors" are a) change the file and b) run the rebuild command - so _me_ as the human who is supposedly keeping an eye on things also take advantage of point 1. _I_ also just have to read the changes to the file and know whats going on.

## the plan

I'm running this experimentally before we commit, but the idea is:
- use Nix to manage our "system underlay"
    - the base ROS installation and standard packages and stuff
- keep colcon as our build tool for _our_ packages in active development
- this will clean up our dev tooling, because we can be free of devcontainers, and rely on more natural tooling that runs right on the dev system
- we get containers back as a development tool (dagger.io vibes to spin up multiple isolated environments for parallel integration testing)
- and in the future, consider moving to NixOS for on-machine deployment (this is also not a "new" idea - there is a fairly active community of projects that use this concept for various embedded devices)

# Skills as a service

_we can make it catchy by turning it into an acronym: SAA-wait, fuck_

ok so the title is related but not the actual topic of today, I just wanted to make that joke. The interesting little thing im thinking about today _is_ skills, but more so how they can be shared in a repo, amongst a team (you can see how i got to "skills as a service", right?). Before we go any further, I would like to disclaim that this is all purely theoretical - I am going to try and set this up, and share skills with other people to see if the idea works, but at the moment im just imagining its a good idea. Oh yeah also, for those who haven't had their entire reality rotted away by AI im talking about [skills](https://code.claude.com/docs/en/skills).

So anyway, whats the idea. Lets start with the premise that there are a lot of things that we do as developers that potentially _only_ we know how to do on the team. For example, im thinking ahead to if/when I roll out this Nix setup for our ROS dev environment, I will by default become the "dev environment fixerer", and unfortunately this covers a very wide surface, from things like intellisense to package management to debugging build failures. I think there is a very nice opportunity here to save myself from having to do a lot of this work, by putting some effort into a bunch of well-crafted skills that can be called upon as a first resort, instead of me. Real quick - what is a skill? Its _bAsIcAlLy_ an extra context file that can be dynamically loaded into a session. It gets loaded two ways - the agent decides its relevant (based on a a "description" of the skill that is made available to it) or the user explicitly references the skill using a "slash command" - typing `/some-skill`.

Lets consider an example - debugging why a build is failing. Now, the first thing we could do is just rely on claude to more or less figure it out itself. I could tell our team, "just say 'hey claude the build is broken, run it and fix any errors'". I tried this, and it _usually_ figures it out, but with a lot of bumbling around figuring out that we are using a nix-managed environment, trying to decide what the right build flags are, then trying to get enough context to come up with a fix etc. Instead, we are going create a `/build-doctor` skill - and in that skill we are going to put a lot of important information about builds, how they work, some common problems, what our expected build commands are etc etc. Note - im not gonna give an actual example of a skill, because who writes skills by hand lol. I'm going to effectively lob a prompt slightly more detailed than the last sentence to claude, and then iterate on whatever it spits out. _(as a side note, i used to hate when I was reading LLM writeups like this and the author was really vague about actual prompts they used - "provide concrete examples dammit" - but im sorry, thats about as concrete as i can be bothered to be, because the real art is in quickly getting *something*, and then iterating it until it sounds good - the key point is you don't have to have all the ideas up front)_

The hope (unverified yet) is that this skill allows users who are less up to speed in a particular area, still have highly productive claude sessions, because I, the person who _is_ up to speed, has made sure that our boy claude knows all the right stuff ahead of time. Lol, it sounds kinda lame now that I type it out, but i think this is quite a powerful concept if done correctly.

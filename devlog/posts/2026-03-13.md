# Harry potter and the case of the linear git history

So this afternoon I was going over a random old PR we had open and in the PR view there was this option like "this branch is out of date with base branch, do you want to update it" and it had options for "merge branch" or "rebase branch". Now, this offended me because I have spent a lot of effort defending our git history from accidental random merge commits. So i initially was just looking for a way to disable that merge branch button.

To give some context - our git strategy up to this point has been merge commits that retain the feature branch history, we just keep our feature branches linear via rebasing. So you end up with the main branch in the middle of the graph, and then you see a feature branch come off main, do some stuff and then merge back into main (or you would if you were using GitKraken instead of being a psycho and rawdogging terminal commands).

Anyway, back to the story - so after some googling (or maybe i asked claude), i determined that what needed to be done here to get rid of that merge option was enable a github "ruleset" (basically a new version of branch protection rules) that targeted all branches except main and sets "Require linear history" to `true`. Thinking about this setting, i was actually quite keen for it as well, because it basically enforces what i have been doing manually, to keep wackass merge commits from absolutely fucking our git history beyond comprehension. So i set up the rule, enabled it... and then i couldnt push any new branches - what the helly, as the kids say.

After some fucking around (i actually dont think claude was able to debug this, but also i think i was relying on its knowlege, should have told it to do some searching) I found this issue (via google like a boomer): https://github.com/orgs/community/discussions/80952. The tldr of this issue is that this option "Require linear history" requires the _entire_ history of the branch to be linear (no merges) not just "new" commits. So every branch that comes of main (which has merge commits) is by definition "non-linear" once its history is traced back to main. (writing this down i think i can see why github has ignored this ticket - how would you define this cleanly? like, commits after the head of main need to be linear? i guess - but kinda tricky to implement). So yeah that sucks - it means i basically can never enforce linear history if my main branch is non-linear (i.e. using merges), because every branch inherits `main`s history. There was some whisperings in that ticket that old "branch protection" rules did allow this old "new linear history only" behaviour, but i tried a branch protection rule and it just didnt seem to enforce anything at all so idk what was going on there.

So now we have two question marks if we wish to continue down the linear history path:
1. are we happy to relinquish "merge" PRs and instead go for either "rebase" or "squash"
2. we need to get our main branch history to be linear

On 1. - "rebase" sounds a bit psycho tbh, because it just puts all of the feature branch commits into the main branch linearly. This seems like hell to try and trace back anything. Squash commits, whilst not idea imo, seem fine - the tradeoff is not having full feature branch history, but also do you really write good commit messages when you're halfway through a feature branch? mine are always shit. (i make sure to write good PRs tho). When weighing up the ability to auto-enforce that our git history doesnt get fucked by merges vs having full feature branch commit history, im leaning towards auto-enforce.

On 2. - in that ticket I found a link to a writeup someone did on figuring out how to do this. https://github.com/orgs/community/discussions/12032#discussioncomment-13465543. Essentially its `git rebase --root` from your main branch and then hold your breath, cross ur toes and `git push origin main --force`.

So yolo, lets re-write two years of repo history, what could go wrong?

I cloned a separate copy of the repo, and ran `git rebase --root`. Basically what this does is it just replays every commit on the main branch in order, to build your history back up. Now, of course this wasnt straightforward - we have some xml files in our repo from PLC programming that i dont want to get into right now, but they have these uuids in them that the plc programming ide will _randomly_ change for no reason and so these xmls always find a way to conflict even though they have very little purpose. _anyway_ so there was a bunch of these conflicting at random points in the rebase process. I did one manually, basically just accepting all the incoming changes, but we had like thousands of commits and it seemed likely that this was going to happen a bit. So thank god for AI - our boy claude was able to whip up a script in no time that ran the rebase and auto accepted incoming changes whenever there was a conflict. The best part was, this is a relatively acceptable thing to do, because at the end of this process i can just diff the head of `main` and `origin/main` and if there are no differences, accepting incoming changes has worked as a strategy - at least for maintaining the correct final code.

For prosperity let me paste in the script claude wrote me:
```
#!/bin/bash

echo "🔁 Starting rebase --root..."

# Start the rebase - will exit non-zero if conflicts, that's ok
git rebase --root || true

# Loop until rebase is complete
while [ -d "$(git rev-parse --git-dir)/rebase-merge" ] || [ -d "$(git rev-parse --git-dir)/rebase-apply" ]; do
    CONFLICTS=$(git diff --name-only --diff-filter=U)

    if [ -n "$CONFLICTS" ]; then
        echo "⚠️  Conflicts detected, auto-accepting incoming changes (--theirs)..."
        echo "$CONFLICTS"
        git checkout --theirs -- .
        git add .
    fi

    echo "▶️  Continuing rebase..."
    GIT_EDITOR=true git rebase --continue || true
done

echo "✅ Rebase complete!"
echo ""
echo "🔍 Verifying result against origin/main..."
DIFF=$(git diff origin/main)

if [ -z "$DIFF" ]; then
    echo "✅ git diff origin/main is empty — code is identical!"
else
    echo "⚠️  Differences detected against origin/main:"
    echo "$DIFF"
fi

echo ""
echo "🔍 Verifying result against origin/main..."
DIFF=$(git diff origin/main)

if [ -z "$DIFF" ]; then
    echo "✅ git diff origin/main is empty — code is identical!"
else
    echo "⚠️  Differences detected against origin/main:"
    echo "$DIFF"
fi
```
